# OVMS Qwen3 0.6B smoke validation

Date: 2026-07-22

This record covers the cache-enabled `v0.1.0` provider image. Compilation
caching is disabled starting with the `v0.1.1` release candidate; see the
follow-up validation below.

This validation exercised the complete local provider image rather than only
the Go gateway in isolation.

## Inputs

| Item | Value |
| --- | --- |
| OVMS base | `openvino/model_server:2026.2-gpu` |
| OVMS digest | `sha256:fdc34725b35d79cde23cff1967f90efb40237b7cf5545a70460209056f3da733` |
| Piccolo AI image | `ghcr.io/atdexters-lab/piccolo-ai-ovms:0.1.0` |
| Model | `OpenVINO/Qwen3-0.6B-int4-ov` |
| Target device | `CPU` |
| Public model alias | `piccolo-chat` |
| Gateway host port | `18000` |
| Direct-backend host port | `18001` |

The model directory was mounted read-only at `/models/model`. The `graph.pbtxt`
created by OVMS pull mode was deliberately removed before startup. OVMS 2026.2
created the graph configuration in memory and did not attempt to mutate the
artifact.

## Results

- Container ran as UID/GID `5000:5000`, inherited from the upstream OVMS image.
- `/healthz` returned `200` after OVMS reported `piccolo-chat` ready.
- Gateway `/v3/models` without a bearer token returned `401 invalid_api_key`.
- Gateway `/v3/models` with the token returned only `piccolo-chat`.
- Direct OVMS `/v3/models` on port 8001 succeeded without a provider token,
  matching the Piccolod private-capability contract.
- Non-streaming `/v3/chat/completions` completed through the gateway.
- Streaming `/v3/chat/completions` delivered incremental SSE events and a final
  `[DONE]` marker.
- `docker stop` produced a graceful OVMS shutdown and container exit code `0`.
- Restarting the same `v0.1.0` container and cache volume returned to ready
  state.
- The `v0.1.0` writable compilation cache contained one blob and used
  approximately 417 MB for this 0.6B INT4 model.

The smoke test published port 8001 only to verify the direct-backend contract.
The Piccolo app will expose that port through a protected capability listener.

## Cache-removal follow-up

Date: 2026-07-23

The `v0.1.1` release candidate was tested on an AMD Ryzen 7 8845HS with the
same model and `CPU` target. Three no-cache launches reached model-specific
readiness in 2.375 s, 1.993 s, and 1.922 s. The corresponding OVMS model-load
phase took 1.684 s, 1.517 s, and 1.501 s.

For comparison, the first cache-enabled launch took 2.742 s and created a
417 MB blob; subsequent warm-cache launches took 0.976-1.035 s. The roughly
one-second warm-start improvement did not justify duplicating almost the full
436 MB model directory for the intended long-running deployment.

The release candidate therefore removes the cache configuration surface and
does not pass `--cache_dir` to OVMS. The rebuilt image had neither
`/var/cache/ovms` nor `/opt/cache`, emitted no cache-enabled startup log, reached
`/healthz` readiness, and returned `piccolo-chat` from authenticated
`/v3/models`.
